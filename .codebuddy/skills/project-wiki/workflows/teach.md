# 工作流：teach（教学式问答）

基于知识库 + 源码，向用户进行教学式解答。与 query 的区别在于**组织信息进行教学**而非简单检索。

## 触发场景

- 用户说"XX 是什么"、"帮我理解 XX"、"讲讲 XX"、"XX 怎么工作的"
- 用户提出了一个技术概念/原理性问题
- 问题本身适合"教学"而非"事实查询"

## 与 query 的区别

| | query | teach |
|---|---|---|
| 目标 | 定位已有信息 | **组织信息进行教学** |
| 回答风格 | 简洁、引用式 | **由浅入深、举例、类比** |
| 是否结合源码 | 可选 | **鼓励**（理论+代码双线） |
| 知识库更新 | 只读 | 发现缺口 → 建议新增文档 |
| 输出长度 | 短（几段） | 中长（结构化教学） |

## 步骤

### 1. 定位相关教程

★ **优先走图谱**（wiki_query.py 一击定位 + 图邻居展开），详见 [query 工作流](./query.md#-推荐路径query-py-一击查询)：

```bash
# 关键词查询（自动展开 alias，输出含 prereq / related / status 警告）
python3 .codebuddy/skills/project-wiki/scripts/wiki_query.py "<关键词>"

# 教学常常需要看一整套教程系列 → 用 series 模式列出全部课程（按 lesson_index 升序）
python3 .codebuddy/skills/project-wiki/scripts/wiki_query.py --series <slug>

# 已知主教程页 → 用 seed 模式展开图邻居（找前置 / 后续 / 同系列课程）
python3 .codebuddy/skills/project-wiki/scripts/wiki_query.py --id 30-tutorials/<slug>/<page>
```

教学场景常涉及"多页综合"（概念 + 机制 + Lyra 实例），属于 query 路由表中**禁止跳过图层**的场景；请重点关注输出里的 `prereq` / `related` / `series-prev/next` 邻居，沿边读 2-3 篇邻居页再组织教学。

只有在 `wiki_query.py` 不可用时，才退回手工三步：
```bash
cat Docs/index.md
rg -l --multiline 'tags:.*<关键词>' Docs/30-tutorials/
rg -i '<关键词>' Docs/30-tutorials/
```

### 2. 判断知识库覆盖度

- **完全覆盖**（教程已有详细内容）→ 基于教程 + 源码组织教学回答
- **部分覆盖**（有概述但缺细节）→ 基于教程 + 补充源码分析回答
- **未覆盖**（知识库无相关内容）→ 基于源码 + 外部资料回答，并建议创建教程

### 3. 组织教学回答

遵循**三层回答结构**：

#### 第一层：概念直觉
- 用**一句话 + 类比**解释核心概念
- 让用户在 10 秒内理解"这是什么"
- 示例："GAS 的 GameplayEffect 就像一个'效果处方单'——它描述了一个效果应该做什么（加血、减速、持续伤害），但本身不执行，需要'应用'到目标身上才生效。"

#### 第二层：技术机制
- 关键代码片段（3-5 行核心逻辑）
- 流程图（mermaid）
- 数据流向
- 关键 API 和参数
- 引用已有教程：`(详见 [[30-tutorials/gas/07-ge-execution-flow]])`

#### 第三层：Lyra 实例
- Lyra 项目中的真实代码使用
- 从 `20-modules/cpp/` 或 `10-architecture/` 引用具体实现
- 说明"在 Lyra 项目中，这个概念被怎样使用"

### 4. 标注引用和缺口

回答中必须：
- 每个技术点标注 `(详见 [[教程页]])`
- 如果知识库完全覆盖，引导用户阅读完整教程
- 如果发现缺口，在回答末尾建议：

```
💡 **知识库建议**：
- 建议为 "XX" 创建教程页（当前未覆盖）
- 建议在 [[30-tutorials/gas/00-gas-overview]] 中补充 XX 相关内容
```

### 5. 可选：crystallize 触发

如果本次教学回答：
- 是一份有结构的深度分析
- 覆盖了知识库的空白
- 用户表示有用、想保留

→ **主动提议**走 **crystallize** 工作流存为新教程页。

## 教学质量检查

回答前自查：
- [ ] 有没有先给概念直觉？（不能上来就贴代码）
- [ ] 有没有配图示？（流程/架构必须有 mermaid）
- [ ] 有没有 Lyra 实例？（理论必须落地）
- [ ] 有没有标注引用？（每个知识点指向教程页）
- [ ] 层次是否清晰？（由浅入深，不跳跃）

## 禁止动作

- ❌ 跳过知识库直接回答（必须先检索）
- ❌ 直接走 `rg` 跳过 `wiki_query.py`（教学是多页综合主场景，会丢 prereq 链 / 漏 ADR）
- ❌ 引用 `status: stale / deprecated` 教程不警告（wiki_query.py 已自动报警，忽略 = 反模式）
- ❌ 直接贴大段代码不解释（代码必须配解读）
- ❌ 回答完不引用教程页（必须建立连接）
- ❌ 发现缺口不提醒（主动建议完善）
