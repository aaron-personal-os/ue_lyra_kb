# 工作流：create-series（创建技术教程系列）

创建一个新的技术专项教程系列，从规划到落地的完整流程。本工作流的核心目标是**产出专业、准确、有深度的技术教程**，而非快速生成表面内容。

## 触发场景

- 用户说"创建 XX 技术专题"、"新增 XX 教程系列"
- 用户提供了一个技术主题，要求系统性文档化

## 与 ingest / digest 的区别

| | ingest | digest | create-series |
|---|---|---|---|
| 输入 | 单篇素材 | 多个已有页 | 一个技术主题 |
| 输出 | 1-2 个 wiki 页 | 1 个综合报告 | **完整系列**（N 篇 + _series.yaml） |
| 目标 | 入库 | 综合分析 | **系统性教学** |
| 源码分析深度 | 浅（提取要点） | 中（综合已有） | **深（必须追到引擎源码）** |

## 质量标杆

本知识库的技术教程不是"概念科普"，而是**基于源码的深度技术分析**。质量标杆：

| 维度 | 不合格 | 合格 | 优秀 |
|------|--------|------|------|
| **准确性** | 凭记忆描述 API | 引用了官方文档 | **从源码逐行验证每个技术断言** |
| **深度** | 只讲"怎么用" | 讲了"怎么实现" | **讲清"为什么这样设计" + 替代方案权衡** |
| **广度** | 只覆盖主流程 | 覆盖了常见分支 | **覆盖边界情况 + 网络同步 + 性能影响** |
| **实践关联** | 纯理论 | 提到了 Lyra 用了什么 | **Lyra 源码逐段解读 + 与引擎默认实现的对比** |
| **图示** | 无 | 有流程图 | **架构图 + 时序图 + 类图按需配合** |

## 调研模式选择

> **在 Phase 0 需求确认时，向用户展示模式选择。默认使用快速模式。**

本工作流支持两种调研模式，主要区别在 **Phase 1（调研阶段）**对源码的分析深度：

| 维度 | 🚀 快速模式（默认） | 🔬 深度解析模式 |
|------|-------------------|----------------|
| **Phase 1 源码分析** | 仅分析关键源码（核心类定义 + 主入口函数） | 深度分析完整调用链 + 所有分支路径 |
| **分析范围** | 关键头文件 + 核心实现段 | 完整 .h/.cpp + 引擎基类 + Lyra 派生类 |
| **SubAgent 使用** | 可选（单 Agent 可完成） | 强烈推荐（多 SubAgent 并行） |
| **外部资料依赖** | 高（官方文档 + Context7 + 社区资源补充细节） | 低（源码为第一手真相，外部资料辅助验证） |
| **Token 消耗** | 低~中 | 高 |
| **速度** | 快 | 慢（2-3x） |
| **准确度来源** | 大模型知识储备 + 外部/内部资料文档 + 关键源码校验 | 源码逐行分析为主 |
| **适用场景** | 概念相对成熟、文档丰富的主题；快速产出初版 | 文档稀少、引擎底层机制、需要极高准确度的主题 |
| **Phase 4 写作时** | 边写边按需查源码验证 | 已有充分调研结果，直接引用 |

### 模式选择时机

在 Phase 0 需求确认时，用 `AskUserQuestion` 让用户选择：

```
模式选择：
- 🚀 快速模式（推荐）— 聚焦关键源码，配合外部资料快速产出，适合大多数主题
- 🔬 深度解析模式 — 完整源码调用链分析，Token 消耗大但准确度最高，适合引擎底层/文档稀缺主题
```

如果用户没有特别指定，默认使用**快速模式**。

## 步骤

### Phase 0：需求确认

向用户确认：
- 主题是什么？（如"UE 渲染管线"）
- 目标读者和前置知识？
- UE 版本？
- 预期篇数和深度？
- 是否有参考资料 / 外部文档 / 引擎源码路径？
- **调研模式**：快速模式（默认）还是深度解析模式？
- **大纲审核**：是否需要在大纲设计后暂停审核？（默认需要，用户明确说"不用审核"/"跳过审核"时可跳过）

---

### Phase 1：调研阶段

> **本阶段决定教程质量的上限。根据所选模式执行不同深度的调研。**

---

#### Phase 1-A：快速模式调研 🚀

> **策略：关键源码校验 + 外部资料补充，平衡效率与准确度。**

##### 1A.1 外部资料优先收集

快速模式下，先收集外部资料建立主题认知框架：

```bash
# 搜索引擎官方文档（首要）
WebSearch "UE5 <topic> documentation site:docs.unrealengine.com"

# 搜索社区最佳实践和深度分析
WebSearch "UE5 <topic> best practices 2025 2026"
WebSearch "UE5 <topic> architecture deep dive"

# 查阅 Context7 文档库
context7 resolve-library-id "Unreal Engine" → query-docs "<topic>"
```

##### 1A.2 关键源码校验

基于外部资料建立的认知，**针对性地验证关键技术点**：

- [ ] **核心类定义**：读取关键头文件（.h），确认类名、继承关系、关键成员
- [ ] **主入口函数**：找到 1-3 个最关键的函数实现，验证核心逻辑流程
- [ ] **Lyra 层对照**：确认 Lyra 中对应类/函数确实存在，快速浏览其定制点

```
分析策略（单 Agent 即可完成）：
1. 根据外部资料提到的核心类，读取其 .h 文件 → 确认类结构
2. 选取 1-3 个最关键的函数，读取实现段 → 验证行为描述的准确性
3. 在 Source/LyraGame/ 中快速定位对应实现 → 确认存在性 + 关键差异
```

##### 1A.3 知识库已有内容检索（必须走图谱）

★ create-series 是"创建新页（系列）"场景，query 路由表中**禁止跳过图层**——否则会与已有教程系列重复或孤立。

```bash
# 主题词 + alias 自动展开
python3 .codebuddy/skills/project-wiki/scripts/wiki_query.py "<topic>"

# 看是否有相邻系列可以挂靠（避免重复造系列）
python3 .codebuddy/skills/project-wiki/scripts/wiki_query.py --series <候选 slug>
```

判定：
- 命中已有系列且高度重合 → 走 **evolve-series** 在该系列加课时，**不**新建系列
- 命中相邻系列 → 把该系列作为新系列的 `prerequisites:` 候选
- 完全无命中 → 才走下文创建新系列

##### 1A.4 快速模式调研产出

调研完成后，整理为：
- **主题认知框架**：核心概念列表 + 它们之间的关系
- **关键源码坐标**：已验证的类名、文件路径、核心函数（后续 Phase 4 写作时按需深入）
- **外部资料索引**：有价值的文档/文章链接及其要点

> ⚠️ 快速模式注意事项：Phase 4 撰写教程时，对每个具体技术断言仍需**按需查阅源码验证**，不能完全依赖外部资料或记忆。快速模式省的是"预先全量分析"的成本，不是"验证"的成本。

---

#### Phase 1-B：深度解析模式调研 🔬

> **策略：源码为第一手真相，外部资料辅助验证。产出完整的技术分析报告。**

##### 1B.1 源码深度分析（核心）

**强烈推荐使用 SubAgent 并行进行源码分析**，因为：
- 单个 Agent 上下文有限，深度源码分析需要读大量文件
- 不同子系统可以并行调研，提高效率
- SubAgent 可以专注于特定层次（引擎层 / Lyra 层）

推荐的 SubAgent 分工策略：

```
主 Agent（规划 + 编排）
    │
    ├── SubAgent-A：引擎层源码分析
    │   - 读取引擎核心类定义和实现
    │   - 梳理类继承关系和关键虚函数
    │   - 追踪核心调用链
    │   - 输出：引擎层架构分析报告
    │
    ├── SubAgent-B：Lyra 层源码分析
    │   - 读取 Lyra 对引擎类的扩展/重写
    │   - 对比与引擎默认实现的差异
    │   - 梳理 Lyra 特有的设计模式
    │   - 输出：Lyra 实现分析报告
    │
    └── SubAgent-C：外部资料收集（辅助验证）
        - WebSearch / Context7 查找官方文档和最佳实践
        - 收集社区常见问题和陷阱
        - 输出：外部资料摘要
```

##### 1B.2 源码分析清单

对于每个教程系列，至少完成以下分析：

- [ ] **核心类继承树**：从引擎基类到 Lyra 派生类的完整继承关系
- [ ] **关键函数调用链**：主要操作的入口 → 中间层 → 底层实现
- [ ] **数据流向**：关键数据结构在各层之间如何传递
- [ ] **网络同步机制**（如适用）：复制策略、预测机制、权威判定
- [ ] **生命周期**：初始化 → 运行 → 销毁的完整流程
- [ ] **Lyra 定制点**：Lyra 在哪些地方重写了引擎默认行为、为什么

##### 1B.3 外部资料辅助验证

```bash
# 搜索引擎官方文档（验证源码分析结论）
WebSearch "UE5 <topic> documentation site:docs.unrealengine.com"

# 搜索社区最佳实践（补充边界情况）
WebSearch "UE5 <topic> best practices 2025 2026"

# 查阅 Context7 文档库
context7 resolve-library-id "Unreal Engine" → query-docs "<topic>"
```

##### 1B.4 知识库已有内容检索（必须走图谱）

同 1A.3，使用 `wiki_query.py` 一击查重：

```bash
python3 .codebuddy/skills/project-wiki/scripts/wiki_query.py "<topic>"
python3 .codebuddy/skills/project-wiki/scripts/wiki_query.py --series <候选 slug>
```

深度模式比快速模式更要重视查重——避免投入大量源码分析后才发现已有覆盖。

##### 1B.5 深度模式调研产出

调研完成后，整理为完整的**技术分析报告**：
- **类继承树图**（mermaid）
- **核心调用链时序图**（mermaid）
- **关键代码段摘录 + 注释**
- **Lyra 定制点清单**
- **设计决策分析**：为什么这样设计 + 替代方案

### Phase 2：大纲设计

基于 Phase 1 的调研结果设计系列大纲。

#### 2.1 结构原则

- **由浅入深**：从概念到实现，从单一模块到系统交互
- **先总后分**：第一篇给全景图，后续逐一深入
- **理论+实践**：每个引擎概念 ↔ Lyra 真实代码
- **独立可读**：每篇有明确的学习目标，可独立阅读（虽然推荐按序）

#### 2.2 文件命名规范

**格式：`序号-中文标题.md`**

文件名直接基于 frontmatter `title` 字段，经以下清洗规则生成：

| 规则 | 说明 | 示例 |
|------|------|------|
| 保留序号前缀 | 两位数字 + 连字符 | `00-`、`01-`、`12-` |
| 保留中英文内容 | 标题中的中文和英文词汇直接保留 | `GAS系统总览`、`Lyra架构总览` |
| 去除空格 | 所有空格（半角/全角）删除 | `GAS 系统` → `GAS系统` |
| 去除特殊符号 | `：:、，。！？；—|·()（）[]【】《》` 等 | `Mutable 是什么：本质` → `Mutable是什么本质` |
| 去除版本号后缀 | `(UE 5.7)`、`（UE5.7）` 等 | `GA输入绑定（UE 5.7）` → `GA输入绑定` |
| 英文缩写括号变连字符 | 括号中纯英文缩写改为 `-缩写` | `GameplayAbility (GA)` → `GameplayAbility-GA` |
| 不使用 `课时N`/`第N课` | 序号前缀已表明顺序，无需重复 | ❌ `01-课时1UObject基础` ✅ `01-UObject基础` |
| 避免过长 | 文件名应简洁概括，不超过 40 字符（不含序号和 .md） | ❌ `05-学习PCG内置节点SurfaceSampler...` ✅ `05-常用PCG节点详解` |

**命名示例：**

```
00-GAS系统总览.md
01-GameplayAbility-GA简介与配置.md
02-GameplayAbility-GA执行流程详解.md
06-ReplicationGraph与Lyra实践.md
04-C++与蓝图交互.md
01-反射是什么从C++到UHT.md
```

**⚠️ 注意事项：**
- `_series.yaml` 中的 `lessons` 数组引用文件名（不含 `.md` 后缀）
- `frontmatter.id` 字段必须与文件路径一致（如 `30-tutorials/gas/00-GAS系统总览`）
- 重命名已有文件必须使用 `rename_page.py` 脚本，确保 wikilink 引用同步更新

#### 2.3 大纲模板

```
00-<系列>系列概览         # 系列概览（必须）：全景图 + 核心概念 + 学习路线
01-<基础概念标题>         # 基础概念：术语 + 分类 + 设计理念
02-<核心机制A标题>        # 核心机制 A：深度源码分析
03-<核心机制B标题>        # 核心机制 B：深度源码分析
04-<核心机制C标题>        # ...
...
NN-1-<高级主题标题>       # 高级主题：性能优化 / 边界情况 / 常见陷阱
NN-Lyra<主题>实战         # Lyra 案例：综合分析 Lyra 的完整实现
```

**示例（GAS 系列）：**
```
00-GAS系统总览.md
01-GameplayAbility-GA简介与配置.md
02-GameplayAbility-GA执行流程详解.md
...
25-Attribute属性详解.md
26-Lyra综合案例死亡能力链.md
```

#### 2.3 每篇预期篇幅

| 类型 | 预期行数 | 说明 |
|------|---------|------|
| 概览（00） | 200-400 | 全景 + 导航，不深入细节 |
| 基础概念 | 300-500 | 概念定义 + 简单示例 |
| 核心机制 | 400-800 | **源码分析是主体**，需要多段代码 + 详细解读 |
| 高级主题 | 300-600 | 性能、网络、边界情况 |
| Lyra 案例 | 400-700 | Lyra 特有实现的完整分析 |

#### 2.4 大纲展示格式

大纲设计完成后，以如下格式展示给用户：

```markdown
## 📋 系列大纲：<系列名称>

**模式**：🚀 快速模式 / 🔬 深度解析模式
**预计篇数**：N 篇
**预计总行数**：XXXX-YYYY 行

| # | 文件名 | 标题 | 核心内容 | 预计行数 |
|---|--------|------|---------|---------|
| 00 | 00-XX系列概览 | ... | ... | 200-400 |
| 01 | 01-基础概念标题 | ... | ... | 300-500 |
| ... | ... | ... | ... | ... |

### 各篇要点预览

**00-XX系列概览**：...
**01-基础概念标题**：...
...
```

### Phase 3：创建骨架文件

#### 3.1 创建 `_series.yaml`

在 `Docs/30-tutorials/<series-slug>/` 下创建：

```yaml
name: 系列名称
slug: series-slug
description: 一句话描述
difficulty: beginner → advanced
ue_version: "5.7"
total_lessons: N
estimated_hours: M
prerequisites:
  - series: ue-framework
    minimum: 00-UE框架概述
  - concept: 前置概念
learning_path:
  - stage: 阶段名
    description: 阶段描述
    lessons: [00-XX系列概览, 01-基础概念]
  - stage: 下一阶段
    lessons: [02-核心机制A, 03-核心机制B]
tags: [tag1, tag2]
```

> **注意**：`lessons` 数组中的值是文件名（不含 `.md` 后缀），必须与实际文件名完全匹配。

#### 3.2 创建概览页（00-XX系列概览）

```markdown
---
id: 30-tutorials/<slug>/00-<系列>系列概览
title: <系列标题>
description: 一句话描述（取自首个引用块）
type: guide
status: current

owner: ai

last_verified: YYYY-MM-DD

lesson_index: 0

---

# 系列标题

> 一句话定位（同时作为 description 字段的来源）

## 概述（是什么、为什么重要）

## 核心概念全景图（mermaid 图示）

## 与 Lyra 项目的关系（映射表）

## 系列阅读指南（分阶段 + 每篇摘要）

## 相关页面
```

> **注意**：`id` 字段必须与文件路径一致（不含 `Docs/` 前缀和 `.md` 后缀）。`title` 字段保留完整标题（含空格、标点），文件名是 title 经清洗规则处理后的结果。

### Phase 4：逐篇撰写教程（核心阶段）

> **每一篇都是独立的深度技术分析，不是走过场。**

#### 4.1 单篇撰写流程

对每一篇教程页：

1. **聚焦源码验证**：
   - **快速模式 🚀**：按需读取与本课相关的源码段，验证关键技术断言。不需要预先全量分析，但每个断言必须有据。
   - **深度模式 🔬**：引用 Phase 1 已有的分析报告，并对本课特定细节做补充验证。
2. **推荐使用 SubAgent 做源码分析**（两种模式均适用，深度模式更需要）：
   ```
   SubAgent 任务：分析 <ClassName> 的实现
   - 读取 Engine/Source/.../ClassName.h 和 .cpp
   - 读取 Source/LyraGame/.../LyraClassName.h 和 .cpp
   - 输出：继承关系、关键函数、调用链、Lyra 定制点
   ```
3. **撰写文档**：基于分析结果撰写，每个技术断言必须有源码依据
4. **配图**：关键流程必须有 mermaid 图

#### 4.2 单篇正文结构

```markdown
## 概述
（一段话：本课学什么、学完能理解什么）

## 核心概念
（概念直觉 → 技术定义 → 分类/分层）
（必须有 mermaid 架构图或分类图）

## 源码深度分析
（本篇的核心价值区域）
```

## 反模式（禁止）

| 反模式 | 为什么不行 | 正确做法 |
|--------|-----------|---------|
| 凭 AI 训练数据记忆写教程 | API 可能已变化，技术细节可能不准确 | **必须读源码验证每个技术断言** |
| 复制外部博客/文档内容 | 版权问题 + 可能过时 + 缺乏 Lyra 关联 | 基于源码的**原创深度分析** |
| 只写概念不贴代码 | 读者无法落地实践 | 每个核心概念配 **引擎源码 + Lyra 源码** |
| 贴大段代码不解读 | 读者看不懂 500 行代码 | **关键行加注释编号 + 逐段解读** |
| 一口气写完不验证 | 容易出现类名拼错、调用链断裂 | **每篇写完 lint + 自检清单** |
| 跳过 Lyra 实践部分 | 丢失"理论→实践"的教学价值 | **每篇必须有 Lyra 代码段** |
| 概览页写得太深 | 吓退初学者 | 概览只给全景图，**深度留给具体课时** |
| 快速模式下完全不看源码 | 准确度无法保证 | 快速模式仍需**按需验证关键断言** |
