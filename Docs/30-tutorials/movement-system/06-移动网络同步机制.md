# 移动网络同步机制

> 深入理解 UE 移动系统的网络同步：客户端预测、服务器校正、以及 Lyra 中的带宽优化方案。

## 概述

UE 的移动网络同步是**客户端预测 + 服务器校正**模型。客户端先"假装"移动成功，再把移动数据发给服务器；服务器验证后，如果不同意客户端的判断，就发送校正包让客户端"回头"。

学完本课你将能够：
- 画出客户端预测与服务器校正的完整时序图
- 解释 `SavedMove_Character` 和 `FSavedMovePtr` 的作用
- 理解 Lyra 的 `FLyraReplicatedAcceleration`（压缩加速度）如何节省带宽
- 在 Lyra 项目中配置和使用 `FastSharedReplication`

---

## 一、网络移动架构总览

### 1.1 核心概念

| 概念 | 说明 |
|--------|------|
| **自治代理（Autonomous Proxy）** | 本地玩家控制的 Character，拥有输入权和预测权 |
| **模拟代理（Simulated Proxy）** | 其他玩家控制的 Character，只能接收网络同步数据 |
| **服务器权威（Server Authority）** | 服务器运行完整物理模拟，验证客户端发来的移动 |
| **客户端预测（Prediction）** | 客户端在发送移动给服务器的同时，先本地执行移动，减少"输入延迟感" |
| **服务器端校正（Correction）** | 服务器发现客户端位置/速度不对时，发送校正包让客户端"回头" |

### 1.2 完整时序图

```mermaid
sequenceDiagram"
    participant Client as 客户端（Autonomous）
    participant Server as 服务器
    participant Other as 其他客户端（Simulated）

    Note over Client,Server: 每一帧（DeltaTime）
    Client->>Client: PerformMovement(DeltaTime)<br/>（本地预测）
    Client->>Server: ReplicateMove(TimeStamp,<br/>Acceleration, CompressedFlags)
    Note over Server: ServerMove() 验证并<br/>执行服务器端移动
    Server->>Server: PerformMovement(DeltaTime)<br/>（服务器权威模拟）
    
    alt 服务器同意客户端位置
        Server-->>Client: Ack（确认收到）
    else 服务器不同意（需要校正）
        Server->>Client: ClientAdjustPosition()<br/>（强制校正位置）
        Client->>Client: ClientRollback()<br/>（回滚到服务器认可的位置）
        Client->>Client: Resimulate()<br/>（重新模拟后续输入）
    end
    
    Note over Server,Other: 同步给其他客户端
    Server->>Other: ReplicatedMovement<br/>（属性复制）或<br/>FastSharedReplication<br/>（快路径快照）
```

## 二、客户端 → 服务器：`ReplicateMoveToServer()`

### 2.1 函数作用

`ReplicateMoveToServer()` 是**客户端向服务器发送移动数据**的核心函数，在 `TickComponent()` 中每帧调用：

```cpp
// Engine/Source/Runtime/Engine/Private/Components/CharacterMovementComponent.cpp:L8789
void UCharacterMovementComponent::ReplicateMoveToServer(float DeltaTime, const FVector& NewAcceleration)
{
    // [1] 检查是否有 PlayerController（只有 Autonomous Proxy 才发送）
    APlayerController* PC = Cast<APlayerController>(CharacterOwner->GetController());
    if (PC == nullptr || PC->Player == nullptr) return;

    // [2] 创建 SavedMove（保存本次移动，用于后续预测回滚）
    FSavedMovePtr NewMove = ClientData->CreateSavedMove();
    NewMove->SetMoveFor(CharacterOwner, DeltaTime, NewAcceleration, *ClientData);

    // [3] 发送到服务器（RPC：ServerMove()）
    PC->ServerMove(TimeStamp, NewAcceleration, CompressedFlags);

    // [4] 保存本次移动到 SavedMoves 列表（等待服务器确认）
    ClientData->SavedMoves.Add(NewMove);
}
```

### 2.2 SavedMove_Character 的作用

`FSavedMove_Character` 保存了一次移动的所有关键信息，用于**服务器校正后的重新模拟**：

```cpp
// Engine/Source/Runtime/Engine/Classes/GameFramework/CharacterMovementComponent.h
class FSavedMove_Character
{
    float TimeStamp;          // 时间戳（用于服务器排序）
    FVector Acceleration;    // 本次输入的加速度
    FVector Location;        // 起始位置
    FRotator Rotation;      // 起始旋转
    uint8 CompressedFlags;   // 压缩的移动标志（Jump、Crouch 等）
    
    // 关键函数
    virtual void SetMoveFor(ACharacter* C, float DeltaTime, FVector Accel, ...);
    virtual void PrepMoveFor(ACharacter* C);  // 重新应用本次移动
};
```

**为什么需要保存？**

当服务器发送校正（说"你位置不对"），客户端需要：
1. 回滚到服务器认可的位置（`ClientRollback()`）
2. 重新应用所有**未被服务器确认的移动**（`SavedMoves` 列表）
3. 这样才能保证"校正后不丢失输入历史"

## 三、服务器 → 客户端：`ClientAdjustPosition()`

### 3.1 什么时候会触发校正？

服务器在 `ServerMove()` 中验证客户端位置，如果发现：
- 客户端位置与服务器模拟位置相差太大（`|ClientPos - ServerPos| > ErrorTolerance`）
- 客户端速度异常（可能作弊，或预测误差累积）
- 客户端穿墙了（服务器检测到碰撞，客户端没检测到）

→ 服务器调用 `ClientAdjustPosition()` 发送校正包。

### 3.2 ClientAdjustPosition() 源码解读

```cpp
// Engine/Source/Runtime/Engine/Private/Components/CharacterMovementComponent.cpp（伪代码）
void UCharacterMovementComponent::ClientAdjustPosition(float TimeStamp, FVector NewLocation, FRotator NewRotation, bool bHasVel, FVector NewVelocity)
{
    // [1] 忽略过时的校正（时间戳旧的）
    if (TimeStamp < ServerLastClientAdjustmentTime) return;

    // [2] 回滚到服务器认可的位置
    ClientRollback();

    // [3] 设置服务器认可的位置/速度
    UpdatedComponent->SetWorldLocationAndRotation(NewLocation, NewRotation);
    Velocity = NewVelocity;

    // [4] 重新模拟所有未被确认的移动
    Resimulate(TimeStamp);
}
```

### 3.3 校正对玩家体验的影响

| 情况 | 玩家看到的现象 |
|--------|---------|
| **小校正**（位置差 < 10 cm） | 几乎无感，角色轻微" snap"一下 |
| **大校正**（位置差 > 100 cm） | 角色突然"瞬移"回正确位置，玩家会感觉"被拽回去了" |
| **高频校正**（每帧都校正） | 角色"颤抖"，网络质量差 |

**优化方向**（Lyra 的做法见第五节）：
- 减少校正频率（`NetworkMaxMoveDeltaTime`）
- 使用 `FastSharedReplication`（ unreliable 快照，允许丢包，不触发大校正）
- 增加 `ErrorTolerance`（允许更大的预测误差）

## 四、属性复制 vs. 快路径快照

### 4.1 默认属性复制：`ReplicatedMovement`

`UCharacterMovementComponent` 默认复制 `FRepMovement` 结构：

```cpp
// Engine/Source/Runtime/Engine/Classes/GameFramework/CharacterMovementComponent.h
struct FRepMovement
{
    FVector_NetQuantize100 Location;  // 位置（精确到 cm）
    FRotator Rotation;                 // 旋转
    FVector_NetQuantize10 Velocity;    // 速度（精确到 0.1 cm/s）
    bool bSimulatedCrouch:1;         // 是否蹲伏
    bool bSimulatedJump:1;           // 是否跳跃
};
```

**问题**：每帧都复制，带宽消耗大（尤其是 60 FPS + 10 人房间 = 600 次/秒）。

### 4.2 Lyra 的优化方案 1：`FLyraReplicatedAcceleration`

Lyra 引入了**压缩加速度**复制（只复制给模拟代理，条件复制）：

```cpp
// Source/LyraGame/Character/LyraCharacter.h:L36-L49
struct FLyraReplicatedAcceleration
{
    uint8 AccelXYRadians = 0;   // 方向（0-255 映射 0-2π）
    uint8 AccelXYMagnitude = 0; // 大小（0-255 映射 0-MaxAcceleration）
    int8 AccelZ = 0;              // Z 轴加速度（-127 到 127）
};

// 在 ALyraCharacter::GetLifetimeReplicatedProps() 中：
DOREPLIFETIME_CONDITION(ALyraCharacter, ReplicatedAcceleration, COND_SimulatedOnly);
```

**带宽对比**：

| 方案 | 每帧字节数（估算） |
|--------|----------------------|
| `FRepMovement`（全量） | ~50 字节/帧 |
| `FLyraReplicatedAcceleration`（压缩） | ~3 字节/帧 |
| 节省 | **94%** |

`OnRep_ReplicatedAcceleration()` 在客户端解压并写入 CMC：

```cpp
// Source/LyraGame/Character/LyraCharacter.cpp
void ALyraCharacter::OnRep_ReplicatedAcceleration()
{
    if (ULyraCharacterMovementComponent* LyraCMC = Cast<ULyraCharacterMovementComponent>(GetCharacterMovement()))
    {
        LyraCMC->SetReplicatedAcceleration(ReplicatedAcceleration);
    }
}
```

### 4.3 Lyra 的优化方案 2：`FSharedRepMovement` + `FastSharedReplication`

这是**更激进的优化**：用 unreliable RPC 发送移动快照，允许丢包（丢包后依赖后续快照恢复）。

#### FSharedRepMovement 结构

```cpp
// Source/LyraGame/Character/LyraCharacter.h:L51-L78
struct FSharedRepMovement
{
    FRepMovement RepMovement;   // 完整的移动数据
    float RepTimeStamp;           // 时间戳
    uint8 RepMovementMode;       // MovementMode（压缩为 1 字节）
    bool bProxyIsJumpForceApplied;
    bool bIsCrouched;
    
    // 自定义网络序列化（只序列化变化的部分）
    bool NetSerialize(FArchive& Ar, class UPackageMap* Map, bool& bOutSuccess);
};
```

#### FastSharedReplication() 触发条件

```cpp
// Source/LyraGame/Character/LyraCharacter.cpp
bool ALyraCharacter::UpdateSharedReplication()
{
    FSharedRepMovement NewData;
    if (NewData.FillForCharacter(this))
    {
        // [1] 与上次发送的数据比较，只有变化时才发送
        if (!NewData.Equals(LastSharedReplication, this))
        {
            // [2] 发送 unreliable Multicast RPC（允许丢包）
            FastSharedReplication(NewData);
            LastSharedReplication = NewData;
            return true;
        }
    }
    return false;
}
```

#### FastSharedReplication RPC（客户端接收）

```cpp
// Source/LyraGame/Character/LyraCharacter.cpp
void ALyraCharacter::FastSharedReplication_Implementation(const FSharedRepMovement& SharedRepMovement)
{
    // [1] 解压 FSharedRepMovement → 写入 CMC
    if (CharacterMovement)
    {
        CharacterMovement->SetRepMovement(SharedRepMovement.RepMovement);
        CharacterMovement->SetMovementMode(SharedRepMovement.RepMovementMode);
        // ...
    }
}
```

**为什么用 unreliable RPC？**

移动快照是"可丢失"的——如果这一帧的快照丢了，下一帧的新快照会覆盖它。用 reliable RPC 反而会因为"重传"导致更严重的延迟。

## 五、Lyra 移动网络同步实战

### 5.1 Lyra 的两种同步路径对比

| 路径 | 触发条件 | 可靠性 | 带宽消耗 | 用途 |
|------|-----------|---------|---------|------|
| `ReplicatedAcceleration` | `COND_SimulatedOnly`（只同步给模拟代理） | Reliable | 低（3 字节/帧） | 模拟代理的加速度显示（用于动画） |
| `FastSharedReplication` | `UpdateSharedReplication()` 检测到变化 | **Unreliable** | 中（~30 字节/帧） | 模拟代理的位置/速度同步 |
| 默认 `ReplicatedMovement` | 未禁用时仍生效 | Reliable | 高（~50 字节/帧） | 权威同步（保底） |

### 5.2 如何在 Lyra 中禁用默认复制，只使用快路径？

```cpp
// 在 ALyraCharacter 构造函数中：
ALyraCharacter::ALyraCharacter(const FObjectInitializer& ObjectInitializer)
    : Super(ObjectInitializer)
{
    // 禁用默认 ReplicatedMovement 复制（改用 FastSharedReplication）
    GetCharacterMovement()->bReplicatesMovement = false;  // [A] 禁用默认复制
    
    // 启用 FastSharedReplication
    GetCharacterMovement()->bEnableFastSharedReplication = true;  // [B] 启用快路径
}
```

**[A]** `bReplicatesMovement = false` 阻止 `FRepMovement` 的默认属性复制。
**[B]** `bEnableFastSharedReplication = true` 让 `UpdateSharedReplication()` 在 `TickComponent()` 中被执行。

### 5.3 常见陷阱：快路径与权威复制冲突

如果**同时启用** `bReplicatesMovement = true` 和 `FastSharedReplication`，客户端会收到**两份移动数据**，导致：
- 位置"抖动"（两路数据不一致）
- 带宽浪费（本来想优化，结果更糟）

**正确做法**：二选一，或只用 `FastSharedReplication`（现代 UE 项目推荐）。

## 六、客户端预测参数调优

### 6.1 关键 CVar

| CVar | 默认值 | 说明 |
|------|---------|------|
| `p.NetEnableMoveCombining` | `1`（启用） | 合并多帧移动为一次 RPC（节省带宽） |
| `p.NetEnableSkipProxyPredictionOnNetUpdate` | `1`（启用） | 收到网络更新时跳过预测（减少计算） |
| `NetworkMaxMoveDeltaTime` | `0.1 s` | 单次移动模拟的最长时间（防止预测爆炸） |
| `NetworkNoSmoothingInServerStates` | `false` | 服务器状态是否禁用平滑（减少校正感） |

### 6.2 Lyra 中的配置

Lyra 在 `DefaultGame.ini` 中配置：

```ini
[/Script/Engine.GameNetworkManager]
NetServerMaxTickRate=30           ; 服务器最高 30 Hz
NetClientTicksPerSecond=60        ; 客户端 60 FPS

[/Script/Engine.CharacterMovementComponent]
NetworkMaxMoveDeltaTime=0.1
NetworkNoSmoothingInServerStates=false
```

## 六、核心要点

在读下一节总结前，确认你理解了以下 5 条：

1. **客户端预测不是外挂** — 所有 `AutonomousProxy` 都先本地执行 `PerformMovement()`，再把结果发给服务器。这是官方设计，不是作弊。
2. **`SavedMove` 是"后悔药"** — 服务器校正时，客户端需要回滚并重新模拟所有未确认的 moves。没有 `SavedMove` 就无法正确处理校正。
3. **`FLyraReplicatedAcceleration` 省带宽的代价是精度** — 方向精度 ~1.4°，加速度精度 ~8 cm/s²（默认 `MaxAcceleration=2048`）。对大部分游戏足够，但竞速类游戏可能需要更高精度。
4. **`FastSharedReplication` 用 unreliable RPC 是故意的** — 移动快照是"可丢失"的，这一帧丢了下一帧会覆盖。用 reliable 反而会因为重传增加延迟。
5. **`bReplicatesMovement=true` + `FastSharedReplication` 同时开启是 Bug 来源** — 模拟代理会收到两份移动数据，导致位置抖动。二选一。

---

## 七、总结

| 要点 | 说明 |
|------|------|
| 客户端预测 | 本地先执行 `PerformMovement()`，再发 `ServerMove()` RPC |
| 服务器校正 | 服务器不同意时调用 `ClientAdjustPosition()`，客户端回滚并重新模拟 |
| 默认复制 | `FRepMovement`（可靠，带宽高） |
| Lyra 优化 1 | `FLyraReplicatedAcceleration`（压缩到 3 字节/帧，只同步加速度） |
| Lyra 优化 2 | `FSharedRepMovement` + `FastSharedReplication`（不可靠 RPC，允许丢包） |
| 最佳实践 | 禁用 `bReplicatesMovement`，只使用 `FastSharedReplication` |
